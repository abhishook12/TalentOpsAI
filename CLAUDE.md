## 5. THE CAMPAIGN ENGINE & OUTLOOK BRIDGE — DEEP ARCHITECTURE

This is the most complex and fragile part of the system. Any AI working on this MUST understand it completely.

### Why the Bridge Exists
Enterprise email accounts (Microsoft 365, Exchange) block third-party SMTP senders. Getting Azure AD admin consent for Microsoft Graph API requires IT department approval that Abhishek doesn't have at Technovion. The bridge is the workaround: it runs locally on his Windows machine and literally controls his desktop Outlook application via COM automation.

### The Email Flow (End-to-End)
```
User creates campaign in browser (Vercel frontend)
    ↓
Frontend POST /campaigns/{id}/start → Render backend
    ↓
send_engine.py: process_campaign_queue() runs as BackgroundTask
    ↓
For each recipient: creates EmailLog with status='sending', sent_via='outlook_bridge'
    ↓
Bridge (local_outlook_bridge.py) polls GET /api/bridge/tasks every 2 seconds
    ↓
Bridge receives task → uses win32com.client to open Outlook → mail.Send()
    ↓
Bridge POSTs results to POST /api/bridge/results
    ↓
Backend updates EmailLog → 'delivered', CampaignRecruiter → 'Sent'
    ↓
When all recipients are terminal → Campaign status → 'completed'
    ↓
Frontend SSE stream (/campaigns/{id}/progress) shows real-time updates
```

### The Timeout Sweep
A background coroutine (`timeout_stuck_emails_sweep`) runs every 15 seconds on the backend. It finds any EmailLog stuck in `sending` status for >30 seconds and marks it as `failed` with error "Outlook Bridge timeout (>30s)". It then increments the retry count on the CampaignRecruiter and either sets status to `retrying` or `failed` depending on whether max retries are exhausted.

## 6. CRITICAL BUGS THAT WERE FIXED — DO NOT REINTRODUCE

### Bug 1: Database Connection Pool Exhaustion via SSE
- **What happened:** The `/campaigns/{id}/progress` SSE endpoint used `Depends(get_db)`, which held a DB connection open for the entire streaming duration. Since SSE streams stay open for minutes, just 20 concurrent viewers would exhaust the 20-connection pool, causing the ENTIRE backend (including bridge polling) to hang with timeouts.
- **Fix:** Removed `Depends(get_db)`. Inside the async generator, use `with SessionLocal() as s_db:` to open/close connections within each polling cycle.
- **Rule:** **NEVER use `Depends(get_db)` on any streaming/SSE/WebSocket endpoint.** Always use short-lived `SessionLocal()` context managers inside the loop.

### Bug 2: Pydantic 422 on Bridge Results
- **What happened:** The bridge was sending `{"log_id": 123, "success": true, "error": null}`. The `BridgeResult` Pydantic model has `error: str = None`. When `error` is explicitly `null` in JSON, Pydantic 2.x sometimes rejects it because it expects either a string or for the field to be omitted.
- **Fix:** In `local_outlook_bridge.py`, only include `"error"` in the payload dict when `error is not None`.
- **Rule:** When constructing bridge result payloads, use: `if error is not None: res_obj["error"] = error`

### Bug 3: Render Cold Starts Breaking Frontend
- **What happened:** Render free tier sleeps the backend after 15 minutes of inactivity. First request takes 30-60 seconds to wake up, causing timeouts.
- **Fix:** The frontend `api.js` has a `smartRequest` wrapper that automatically retries on 502/503/504 errors with a 1.2-second delay. The UI shows loading states.
- **Rule:** Always ensure any new frontend API calls go through the `api` wrapper from `services/api.js`, never raw `axios` or `fetch`.

## 8. AUTHENTICATION ARCHITECTURE

The auth system has TWO layers that coexist (for historical reasons):

### Layer 1: Modern JWT Auth (Primary)
- User registers → password validated (8+ chars, uppercase, lowercase, digit, special char) → bcrypt hash → stored in `users` table
- Login → returns JWT access token + refresh token (set as httpOnly cookies AND returned in response body)
- Frontend stores token in `localStorage`/`sessionStorage` via `setStoredToken()`
- Every API request adds `Authorization: Bearer <token>` header via the `api.js` wrapper
- `/auth/me` validates the token and returns user info
- Sessions tracked in `db_sessions` table (allows remote logout)

### Layer 2: Legacy Admin Auth (Still Active)
- Simple password-based admin login at `/auth/admin-login`
- Returns a JWT with `{"role": "admin"}` payload
- Stored in `admin_session` cookie
- Still functional — the AdminTerminal page works with either auth method

### Admin Detection (Frontend)
```javascript
const isAdmin = user?.email?.toLowerCase() === 'abhishekjadon824@gmail.com' || 
                ['admin', 'superadmin'].includes(user?.role?.toLowerCase() || '');
```
Abhishek's personal email is hardcoded as always-admin. This is intentional.

### RBAC Roles
`superadmin` > `admin` > `manager` > `recruiter` > `user` > `readonly`

### Google OAuth
- Supported via `/auth/google` endpoint
- Accepts Google ID token, verifies it server-side
- Auto-creates user if first login, or links to existing account
- Sets `auth_provider` to `"google"` or `"both"`

## 9. THE RESOURCE LOCKDOWN SYSTEM

The backend has an automated emergency lockdown system (`resource_lockdown.py`) that monitors:
- **Database size:** Triggers lockdown at 400MB (Supabase free tier limit)
- **Server memory:** Triggers at 250MB
- **Gemini API rate:** Triggers at 7 RPM (70% of free-tier 10 RPM limit)

When locked down, ALL non-GET requests return 503 with `"System is in Emergency Lockdown"`. This is checked via middleware on every request.

## 11. THE SMART IMPORT SYSTEM

The data import pipeline is one of the most sophisticated parts:
1. User uploads CSV/Excel → `adaptive_parser.py` uses Gemini AI to detect column meanings
2. `format_detector.py` identifies the file format and structure
3. `import_service.py` orchestrates: column mapping → validation → deduplication → enrichment → insert
4. `dedup_engine.py` does fuzzy matching on email, phone, and name (Levenshtein distance + phonetic matching)
5. Records needing human review go to the Review Queue (`/review-queue`)
6. The entire process streams progress via SSE so the frontend shows a real-time progress bar

## 15. KNOWN QUIRKS AND GOTCHAS

1. **PowerShell on Windows:** Use `;` not `&&` to chain commands. Use `Get-Content` not `cat`. Unicode emojis in Python print statements cause `UnicodeEncodeError` — use `PYTHONIOENCODING=utf-8` or avoid emojis.

2. **The `recruiters` table has 30+ columns** including multiple alternate email/phone fields, trust scores, data source tracking, location confidence, and completeness scores. The ORM model is in `models/models.py` and startup migrations in `main.py` dynamically add columns if missing.

3. **`AdminTerminal.jsx` is 128KB** — it's a massive single-file component. Any AI editing it should be very careful about context window limits.

4. **The frontend uses TanStack Router, NOT React Router.** Routes are defined in `frontend/src/router.jsx` using `createRouter`, `createRoute`, `createRootRoute`. Do NOT import from `react-router-dom`.

5. **CORS:** The backend allows specific origins + a regex for `*.vercel.app`. If you add a new frontend domain, update `CORS_ORIGINS` in `config.py` or the Render environment.

6. **The `from_email` on campaigns** is stored but the bridge uses whatever Outlook account is configured locally. The bridge doesn't switch sender accounts — it sends from whichever account is the default in the local Outlook installation.

7. **`OmniRoute/`** is a git submodule of an open-source project. It is NOT part of TalentOpsAI core. Ignore it when working on TalentOps features.

8. **The `activate_user.py` script** directly connects to Supabase (hardcoded connection string) to set a user's status to "Active", bypassing email verification. This is a dev/testing utility.
